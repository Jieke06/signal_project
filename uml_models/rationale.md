# CHMS Subsystem Design Rationales

## 1. Alert Generation System Rationale
This subsystem utilizes the Strategy Design Pattern to separate data evaluation logic from the generic alert triggering framework. By encapsulating distinct clinical evaluation behaviors within standalone classes like HeartRateStrategy and BloodPressureStrategy, the subsystem adheres to the Open-Closed Principle. This modular design permits the seamless addition of novel medical rules without breaking existing functionality. Personalized clinical criteria are securely isolated inside patient-specific Threshold entities. Upon detecting vital anomalies, the system instantly instantiates an Alert object and routes it toward the AlertManager. The manager serves as a dispatcher, controlling routing channels to guarantee that time-critical notifications are dispatched efficiently to the respective healthcare authorities.

## 2. Data Storage System Rationale
The design of the Data Storage Layer focuses heavily on security, privacy, and long-term maintainability. Instead of exposing raw data maps to peripheral system components, it enforces strict access control mechanisms through the SecurityController. This entity implements authorization verifications before permitting medical data retrieval, meeting safety-critical hospital privacy regulations. The inclusion of the RetentionPolicyManager addresses data lifecycle management by providing automated routines to purge expired patient records dynamically, which effectively prevents storage degradation. Patient records are wrapped inside a versioned PatientData model, ensuring historical traceability and full data integrity during retrospective clinical diagnostic analysis.

## 3. Patient Identification System Rationale
Patient traceability and data-matching integrity represent core pillars of safety-critical systems. The Patient Identification Subsystem serves as an abstraction bridge connecting raw simulator streams to formal internal structures. The central coordinator, IdentityManager, relies on the PatientIdentifier to map primitive incoming indices to authenticated HospitalPatient objects securely. To account for potential boundary anomalies—such as data streams arriving with unregistered IDs—the system incorporates a decoupled MismatchHandler. Rather than allowing an unmapped packet to cause a fatal application crash, the system logs the incident, quarantines the data record, and schedules a safe, manual review protocol, ensuring high system resilience.

## 4. Data Access Layer Rationale
The Data Access Layer establishes a robust interface structure designed to isolate external ingestion frameworks from the application's underlying business domain. By creating the abstract DataListener foundation and deriving protocol-specific subclasses, the network layer ensures that the core tracking program remains oblivious to whether the metrics arrive via live TCP packets, active WebSockets, or static file logs. Raw data variations are standardized using polymorphic DataParser components. Ultimately, parsed objects are picked up by the DataSourceAdapter, which normalizes incoming values before persisting them safely inside the storage module, maximizing separation of concerns.
